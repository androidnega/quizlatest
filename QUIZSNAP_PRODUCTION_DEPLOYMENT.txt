1.3  public/.htaccess — HTTPS enforcement add-on (recommended)
──────────────────────────────────────────────────────────────────────────────

The current public/.htaccess (95 lines) handles:
  • PATH_INFO rewriting to index.php                                ✓
  • Authorization + XSRF header forwarding                          ✓
  • Trailing-slash redirect                                         ✓
  • Browser cache (1 year for hashed Vite assets)                   ✓
  • mod_deflate gzip for JSON / HTML / JS / CSS / font / SVG        ✓
  • Set-Cookie stripping on static assets                           ✓

What it does NOT yet enforce: HTTPS at the Apache layer. cPanel's
"Force HTTPS Redirect" toggle is the supported path. If for any reason
that toggle is unavailable, add the following block to public/.htaccess
immediately after the `RewriteEngine On` line (line 6):

    # Force HTTPS at the edge (used only if cPanel "Force HTTPS" toggle is off).
    RewriteCond %{HTTPS} !=on
    RewriteCond %{HTTP:X-Forwarded-Proto} !=https
    RewriteRule ^ https://%{HTTP_HOST}%{REQUEST_URI} [L,R=301]

Also recommended (a one-liner security baseline appropriate for an
education tenancy):

    <IfModule mod_headers.c>
      Header always set Strict-Transport-Security "max-age=31536000; includeSubDomains"
      Header always set X-Content-Type-Options "nosniff"
      Header always set Referrer-Policy "strict-origin-when-cross-origin"
      Header always set X-Frame-Options "SAMEORIGIN"
    </IfModule>

These are infrastructure hardening, NOT new features — they only ride on
top of what Apache already serves.

2.6  Rollback safety
──────────────────────────────────────────────────────────────────────────────

  • Every recent (2026-06-*) migration uses `Schema::hasTable`,
    `Schema::hasColumn`, and try/catch wrappers — re-runs are idempotent.
  • Bulk-data backfill steps (e.g. 2026-06-09 last_violation_at
    backfill) update only rows matching `whereNull(...)`, so a partial
    re-run resumes without duplication.
  • The violation_events JSON column is RETAINED for back-compat — the
    migration comment explicitly notes a future sprint will drop it.
    Today, every reader/writer path tolerates the column being present
    or null.
  • `down()` paths exist for all 2026-06-* migrations — but they should
    NOT be used in production; rollback strategy is "roll forward" via
    a new corrective migration plus restore-from-backup if data was
    affected (see Phase 4).

4.2  Backup schedule
──────────────────────────────────────────────────────────────────────────────

  Schedule::command('qs:backup:run', ['--keep-days=7'])
      ->dailyAt('02:00')
      ->withoutOverlapping(120)
      ->name('qs.backup.daily');

    institutional time zones).
  - withoutOverlapping(120) means if yesterday's run is still mid-tar at
    02:00 today, today's run skips (2-hour lock) — protects against
    runaway dumps on a very large DB.

       cd ~/quizsnap_app
       gunzip < storage/app/backups/db-2026-06-09-0200.sql.gz \
         | mysql -u USER -p DB

     mysqldump output includes CREATE TABLE statements, so this
     reconstructs everything in one shot.

  B. Storage restore:

       cd ~/quizsnap_app
       mv storage/app storage/app.bad-$(date +%FT%H%M)
       tar -xzf storage/app.bad-*/backups/files-2026-06-09-0200.tar.gz
       # verify, then rm -rf storage/app.bad-*

  C. Single-table emergency restore (fastest during a live exam):

       zcat storage/app/backups/db-LATEST.sql.gz \
         | sed -n '/^-- Table structure for `exam_sessions`/,/^-- Table/p' \
         > /tmp/exam_sessions_only.sql
       mysql -u USER -p DB < /tmp/exam_sessions_only.sql

  D. Code rollback:

       git fetch --tags
       git reset --hard <last-known-good-tag>
       composer install --no-dev --optimize-autoloader
       npm ci && npm run build
       php artisan config:clear && php artisan view:clear
       php artisan migrate --force        # idempotent — safe
