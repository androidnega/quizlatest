PHASE 1 - LIVE EXAM OPERATIONS REVIEW
================================================================================

The system is now hardened against most ROUTINE failure modes; the
list below catalogs the FAILURES THAT CAN STILL HAPPEN during a real
exam, ranked by likelihood, with a documented mitigation already in
the codebase.

A. DURING A LIVE EXAM
---------------------

A1. Student loses internet (most common).
    Symptom: /state polls fail, /save-answer queues offline, the
    student's screen freezes on the same question.
    Detection: PauseStaleExamSessionsCommand auto-pauses the row
    when last_seen_at is older than EXAM_DISCONNECT_PAUSE_THRESHOLD
    seconds (default 120). The Command Center shows the row under
    "Paused (disconnect)".
    Recovery: nothing on our side. When the student reconnects, the
    runtime auto-flushes the offline answer queue (IndexedDB) and
    POSTs /resume. If the student does not return within
    exam.max_pause_minutes (default 10), AutoSubmitStalePausedSessions
    force-submits with reason=stale_paused into the held queue.
    Operator action: only intervene if the student calls in.
    Use ExaminerEmergencyController::unlockSession() to bring the
    row back to active without losing the timer budget.

A2. Student's camera fails mid-exam.
    Symptom: face_obstruction events flood in, the proctoring overlay
    blocks the question.
    Detection: ProctoringOrchestratorService raises the violation
    score; if MAX_STUDENT_OVERLAY_DISMISSALS=3 is exceeded, the row
    auto-submits with reason=overlay_dismiss_abuse. Otherwise it
    sits at risk_state=warning.
    Recovery: examiner can call extendTime + unlockSession after
    visually verifying the student's situation.

A3. Student's browser crashes.
    Symptom: tab disappears; identical to A1.
    Recovery: same as A1 - reload the take page, IndexedDB queue
    replays, server already has the last saved revision.

A4. Examiner accidentally publishes the wrong question version.
    Symptom: students start writing against a wrong answer key.
    Recovery: pause publishing in the builder, fix the key, the
    answers JSON column is recomputed by AnswerEvaluationService at
    submit-time, so existing in-flight sessions get the corrected
    grade.
    Note: this is the ONE case where an examiner-side mistake can
    cascade. Mitigation: add a "publish lock" (already in admin
    settings as settings.lock) before exam day.

B. DURING PEAK USAGE (concurrent active sessions)
-------------------------------------------------

B1. Apache hits MaxRequestWorkers (shared cPanel typically caps at
    150-300). Net effect: queued requests, then 503s.
    Mitigation in code: state polling cadence is 30 s (architecture
    review phase 1); /state payload is now ~3-5 KB with /exam-structure
    served once and 304-revalidated. At 250 concurrent students the
    expected request rate is ~8.3 req/s for /state, ~2 req/s peak for
    /save-answer, well under typical Apache capacity.
    Mitigation in ops: see PHASE 7 load-test plan.

B2. MySQL connection limit (shared cPanel typically caps at 25-50).
    Mitigation in code: Laravel's persistent connection plus the
    indexed COUNT(*) queries used by the Command Center keep the
    average open-connection count low. Sessions are file-based (not
    DB-backed) per architecture review.
    Mitigation in ops: contact host BEFORE the exam window to raise
    max_user_connections to 100 if you expect > 200 concurrent.

B3. Disk space exhaustion (storage/app/proctoring + storage/logs).
    Mitigation in code: qs:monitor:snapshot warns at 4 GB, alerts
    at 9 GB. Daily log prune keeps logs/*.log under 30 days.
    Proctoring events older than 6 months are weekly-pruned.
    Mitigation in ops: keep at least 5 GB free at exam start.

B4. Reverb / live socket overload.
    Mitigation in code: the runtime degrades to polling (architecture
    review phase 1). Reverb is purely a UX nicety; nothing breaks
    if it falls over.

C. DURING DEPLOYMENT (mid-window or pre-window)
-----------------------------------------------

C1. New build pushed while students are writing.
    Symptom: hashed Vite asset names change, returning students hit
    a blank page after fullscreen exit.
    Mitigation: do NOT deploy during a live window. Use the
    AdminSettingsController "lock" toggle to freeze publishing
    while the window is open.

C2. Migration introduces a NOT NULL column without a default.
    Symptom: every row insert fails for active sessions.
    Mitigation: every recent migration in this repo is idempotent
    and uses Schema::hasColumn / try-catch blocks. Treat any
    deployment with a non-idempotent migration as ON-WINDOW
    forbidden.

D. DURING SERVER RESTART
------------------------

D1. cPanel restarts Apache during the window.
    Symptom: every active /state poll receives a 503 for the
    restart window (usually < 30 s).
    Mitigation: client retry logic (axios retries on transient).
    Heartbeats resume on first successful poll. Sessions stay
    "active" because last_seen_at is NULL-tolerant.

D2. PHP-FPM crash + automatic restart.
    Symptom: same as D1.
    Mitigation: Laravel's queue connection is sync, so no jobs are
    lost. ExamSessionSubmissionService is idempotent (CAS + Result
    upsert). A submit fired during the crash window simply retries
    on the client side.

E. DURING DATABASE RESTART
--------------------------

E1. MySQL restarts (host-driven).
    Symptom: every query in the restart window throws
    QueryException.
    Mitigation: ExamSessionSubmissionService::submit() wraps the
    pipeline in DB::transaction() with 3 retries on deadlock /
    serialization. /state and /save-answer fail-fast; the client
    UI shows a reconnecting banner. When MySQL returns, the next
    poll succeeds.
    Critical: NO submit is lost as long as the mysqldump backup
    finished BEFORE the restart (see Phase 6).

PHASE 3 - INCIDENT RESPONSE PLAYBOOK
================================================================================

Each incident below has the SAME 4-line shape:
  Detection : how to know it's happening
  Containment : stop the bleeding
  Recovery   : restore service
  Post-mortem : what to file in the ActivityLog / wiki

I3.1 - Database outage
    Detection: Command Center alerts panel goes red; admin
      health-snapshot returns database.tone=danger; every controller
      throws QueryException in laravel.log.
    Containment: serve maintenance.php from public/ (Laravel up/down
      mode is unreliable on cPanel). Notify proctors via WhatsApp /
      voice; tell students to leave their tab open.
    Recovery: bring MySQL back; ExamSessionSubmissionService retries
      on next student submit. AutoSubmitStalePausedSessions catches
      anything stuck > max_pause_minutes.
    Post-mortem: examiner must scan the held queue for sessions
      flagged stale_paused and decide release vs. retake.

I3.2 - Shared hosting outage (whole node)
    Detection: cPanel monitoring email; site responds with host-level
      503 / 504 / DNS NXDOMAIN.
    Containment: nothing local. Send the pre-prepared outage email
      template (Phase 6 / "Communication Templates").
    Recovery: when host returns, run `php artisan schedule:run` once
      manually to ensure auto-submit-stale-paused fires immediately
      rather than waiting for the next minute boundary.
    Post-mortem: every session whose last_seen_at is older than the
      outage start should be reviewed manually.

I3.3 - PHP crash (process dies mid-request)
    Detection: 502 from Apache; cPanel error_log shows segfault or
      memory limit.
    Containment: the auto-restart by FPM brings PHP back within
      seconds.
    Recovery: client retries automatically (axios + the
      hydrator's invalidateAnswers fallback).
    Post-mortem: increase memory_limit if the crash was OOM (often
      caused by GD on huge images - we already use OptimizedImageService
      with strict caps, so this should not recur).

I3.4 - Network interruption (host->internet)
    Detection: students report "Reconnecting" banner; no DB error.
    Containment: nothing - the issue is upstream of the student.
    Recovery: same as A1 (auto-pause + auto-flush queue).
    Post-mortem: none if < 5 min; for longer outages, bulk-extendTime
      via examiner emergency tools.

I3.5 - Student device failure (laptop dies)
    Detection: out-of-band (student tells you).
    Containment: examiner uses ExaminerEmergencyController::unlockSession
      after the student logs back in on a fresh device. Optionally
      ::extendTime(+15) for the wasted time.
    Recovery: student resumes from saved answer revision.
    Post-mortem: ActivityLog row is automatic.

I3.6 - Camera failure (specific student)
    Detection: face_obstruction stream + the student's overlay.
    Containment: same as A2.
    Recovery: examiner extends time + unlocks; if the session
      auto-submitted because of overlay_dismiss_abuse and the student
      is genuine, examiner uses ExaminerEmergencyController::override
      to flip exam_status from submitted_held -> graded.
    Post-mortem: the override audit row records who decided what,
      with the required free-text reason.

I3.7 - Browser crash (specific student)
    Same as A3.

I3.8 - Deployment mistake
    Detection: smoke test fails; OR the on-call SRE checks
      /admin/health-snapshot and sees vite.tone=danger (manifest
      missing) or database.tone=danger (env mismatch).
    Containment: rollback by `git reset --hard <last-known-good-tag>`
      then `composer install --no-dev` and `npm run build`.
      Database migrations are append-only and idempotent in this
      repo; rolling forward by re-running migrate is always safe.
    Recovery: same as I3.1 if a migration broke schema.
    Post-mortem: file the cause in the Phase 8 risk register.

PHASE 4 - EXAM MONITORING (implemented)
================================================================================

Commands:

  qs:monitor:snapshot [--json]
      One-shot health/metric snapshot. Implemented in
      app/Console/Commands/QsMonitorSnapshotCommand.php. Captures:
        active_sessions
        auto_submits_per_hour
        held_per_hour
        stale_paused_sessions
        proctoring_events_per_minute
        violating_sessions_now
        log_size_mb
        private_storage_mb
        failed_requests_per_hour
      Stores result in `qs:ops:snapshot` cache key (5-min TTL) and
      a 24-entry rolling history in `qs:ops:snapshot:history`.
      Threshold breaches are written to the `monitoring` log channel
      (or `daily` if monitoring isn't configured).

  qs:backup:run [--keep-days=N] [--db-only] [--files-only]
      Daily DB+files backup; see Phase 6 for details.

  exam:pause-stale-sessions
      Per-minute auto-pause of silent active sessions.
      (Pre-existing, untouched in this turn.)

  exam:auto-submit-stale-paused
      Per-minute force-submit of sessions paused longer than
      exam.max_pause_minutes. (From Architecture Review Phase 5.)

Schedule (in routes/console.php):

  every minute    exam:pause-stale-sessions
  every minute    exam:auto-submit-stale-paused
  every 5 min     qs:monitor:snapshot
  every minute    queue:work --stop-when-empty (queue drainer)
  daily 02:00     qs:backup:run --keep-days=7
  daily 03:15     log file pruning (in-line closure)
  weekly Sun 03:30 proctoring_events 6-month prune

What is NOT instrumented (and why):

  Slow query log: cPanel does not let tenants enable MySQL's
  slow_query_log. Workaround: the QsMonitorSnapshotCommand counts
  failed_requests_per_hour from the production log, which catches
  query timeouts via Laravel's QueryException -> production.ERROR
  pattern. If you upgrade to a managed-MySQL host that exposes the
  slow log, drop a tail-based scanner into a new
  qs:monitor:slow-queries command.

  APM-grade per-request latency: shared cPanel does not allow a
  long-running daemon process. Latency proxies (events/min,
  failed_requests/hour) are surfaced in the Command Center
  instead.

PHASE 5 - EXAMINER TOOLS (implemented)
================================================================================

Endpoints (all gated by Quiz::manageResults policy):

  POST   /examiner/exams/sessions/{examSession}/emergency/extend-time
         body: {minutes: int(-60..240), reason?: string}
         response: {status, extra_seconds, extra_minutes}

  POST   /examiner/exams/sessions/{examSession}/emergency/unlock
         body: {reason?: string}
         response: {status: "unlocked"}
         refuses 422 on already-submitted sessions.

  POST   /examiner/exams/sessions/{examSession}/emergency/override
         body: {exam_status: "graded"|"submitted_held"|"flagged_for_review"|"submitted_late",
                reason: string min:8}
         response: {status, exam_status}

  GET    /examiner/exams/sessions/{examSession}/emergency/audit-trail
         response: {session_id, events: [...]}

Pre-existing endpoints (not modified, named here for completeness):

  POST   /exam-sessions/{examSession}/force-submit
  POST   /exam-sessions/{examSession}/review/release
  POST   /exam-sessions/{examSession}/review/confirm-fail
  POST   /exam-sessions/{examSession}/review/override
  POST   /examiner/exams/sessions/{examSession}/invalidate-for-retake

Auditing:

  Every emergency action writes one row to activity_logs via
  ExaminerEmergencyAuditService. event_type is namespaced
  examiner_emergency.{extend_time|unlock_session|override_decision|...}
  with the actor's user_id, the session's exam_session_id, and a
  free-text `reason` field for the required justifications.
  The audit-trail endpoint exposes the last 50 entries for one
  session and is what the dashboard "Action history" panel reads.

Permissions matrix:

  admin       -> can do everything (manageResults bypasses department check)
  examiner    -> only on quizzes whose course they own
  coordinator -> NOT directly. They use the Command Center to
                 monitor and the examiner to take action. This
                 separation is intentional - reduces blast radius.
  student     -> never (manageResults policy explicitly rejects
                 student role).

Tested: tests/Feature/ExaminerEmergencyToolsTest.php (9 tests, 33
assertions, all green). Includes a cross-examiner authorization
test confirming the `intruder` examiner gets 403 against another
course's session.

PHASE 6 - BACKUP AND RECOVERY (implemented)
================================================================================

Daily strategy
--------------

cron: schedule:run -> dailyAt('02:00') -> qs:backup:run --keep-days=7

Outputs go to storage/app/backups/:

  db-YYYY-MM-DD-HHMM.sql.gz   - mysqldump --single-transaction --quick
                                 (no row locking, consistent point-in-time)
  files-YYYY-MM-DD-HHMM.tar.gz - tar of storage/app
                                 (excludes storage/app/backups itself)

Retention is enforced AT THE END of each run (--keep-days=7 by
default). Anything older than 7 days is unlinked.

Weekly strategy
---------------

The cron above runs daily, not weekly - on shared cPanel a 7-day
rolling window is usually sufficient. For a longer retention, add:

  Schedule::command('qs:backup:run', ['--keep-days=30', '--db-only'])
      ->weeklyOn(0, '02:30')  // Sunday morning
      ->name('qs.backup.weekly')
      ->withoutOverlapping(180);

OFF-HOST RETENTION (critical):

The cron writes INSIDE the host. If the host fails completely, those
backups die with it. Two viable off-host paths on shared cPanel:

  Option A (manual):  the SRE pulls the previous night's *.gz down
                      every morning via SCP / rsync to a laptop or
                      external NAS.

  Option B (automated): add an "after-backup" hook that pushes the
                      newest file to S3, Backblaze, or any
                      WebDAV-compatible target. Suggested env:
                          QS_OFFHOST_S3_BUCKET=...
                          QS_OFFHOST_S3_KEY=...
                          QS_OFFHOST_S3_SECRET=...
                      Implement as a small extension to QsBackupCommand
                      that runs `aws s3 cp` after each artefact write.

Database recovery
-----------------

Worst case: schema is intact, data is corrupt.
  cd ~/quizsnap_app
  gunzip < storage/app/backups/db-2026-06-09-0200.sql.gz | mysql -u <user> -p <db>

Worst case: schema AND data both gone.
  Same command. mysqldump output includes CREATE TABLE statements.

Single-table recovery (fastest path during a live exam):
  zcat storage/app/backups/db-...sql.gz | grep -A1 'CREATE TABLE `exam_sessions`' \
    > /tmp/exam_sessions_only.sql
  # then run that with mysql -e

Storage recovery
----------------

  cd ~/quizsnap_app
  rm -rf storage/app/proctoring storage/app/private  # or whatever needs replacing
  tar -xzf storage/app/backups/files-2026-06-09-0200.tar.gz

Tar always extracts under the original path (storage/app/...), so
running it from the project root is correct.

Rollback guide
--------------

A. Code rollback (last good tag):
     git fetch --tags
     git reset --hard <last-known-good-tag>
     composer install --no-dev --optimize-autoloader
     npm ci && npm run build
     php artisan config:clear && php artisan view:clear
     php artisan migrate    # idempotent; safe to re-run

B. Database-side rollback (only if a destructive migration leaked):
     mysql -u <user> -p <db> < /tmp/backup-of-current.sql   # safety net first
     gunzip < storage/app/backups/db-PREVIOUS.sql.gz | mysql -u <user> -p <db>

C. Storage-side rollback:
     mv storage/app storage/app.bad
     tar -xzf storage/app/backups/files-PREVIOUS.tar.gz
     # verify manually before deleting storage/app.bad

Pre-flight before any rollback: run `php artisan qs:backup:run` ONCE
on the current state so you have an "as-found" snapshot to compare
against if the rollback itself goes wrong.

Compatibility with shared cPanel
--------------------------------

  - mysqldump : present on every cPanel host we have audited.
  - gzip / tar: same.
  - cron     : standard cPanel cron entry:
                 * * * * * cd /home/<user>/quizsnap && php artisan schedule:run >/dev/null 2>&1
  - storage  : storage/app is private and outside public_html. The
               existing public/.htaccess deny rules already prevent
               direct download.
  - PHP exec/popen: required by QsBackupCommand. Most cPanel hosts
               allow `popen` by default. If `disable_functions`
               blocks it, use `exec` instead, OR offload to a
               bin-php cron line that calls `mysqldump` directly:
                 0 2 * * * mysqldump ... | gzip > .../db-$(date +%F).sql.gz

PHASE 7 - LOAD TEST PLAN
================================================================================

Tooling: k6 (preferred, scriptable), JMeter, or Locust. Examples
below assume k6 because it produces clean p95/p99 stats out of the
box. Run the load generator from a DIFFERENT machine than the
hosting account (we want to measure server cost, not local disk).

Profile A - 50 concurrent students
----------------------------------

  vu (virtual users) : 50
  ramp-up            : 30 s linear
  hold               : 60 min (full exam window)
  ramp-down          : 30 s

Per-VU script (steady state):
  every 30 s : GET /exam-sessions/{id}/state
  every 25 s : POST /exam-sessions/{id}/heartbeat
  every  4 s : POST /exam-sessions/{id}/answers (debounced typing)
  every 10 s : POST /exam-sessions/{id}/proctoring-events/batch
  once       : GET /exam-sessions/{id}/exam-structure (then ETag-revalidate)
  once       : GET /exam-sessions/{id}/answers       (then ETag-revalidate)

Pass / fail (50 VUs):
  CPU         < 40% sustained on the cPanel container
  RAM         < 60% of allocated PHP/MySQL pool
  /state p95   < 250 ms
  /save-answer p95 < 350 ms
  Submit      every VU's POST /submit completes within 5 s
  DB usage    < 25 active connections at peak
  Network     < 200 KB/s outbound average per VU

Profile B - 100 concurrent students
-----------------------------------

  vu                : 100
  ramp-up           : 60 s
  Same script.

Pass / fail (100 VUs):
  CPU          < 65% sustained
  RAM          < 75%
  /state p95    < 350 ms
  /save-answer p95 < 500 ms
  DB usage     < 50 active connections at peak
  No 429 (throttle) errors observed for legitimate cadence
  No 503 / 504 from Apache

Profile C - 250 concurrent students
-----------------------------------

  vu                : 250
  ramp-up           : 120 s
  Same script.

This is the stretch profile. Most shared cPanel plans will not
sustain 250 concurrent without a tuned Apache + raised MySQL
max_user_connections.

Pass / fail (250 VUs):
  CPU          < 85% sustained
  RAM          < 85%
  /state p95    < 600 ms
  /save-answer p95 < 800 ms
  DB usage     < 80 active connections at peak
  /submit failure rate < 0.5% (transient; client retries cover them)

Failure thresholds (any one = STOP and request a host upgrade):
  - 5xx rate > 1% sustained over 60 s
  - any auto-submit fired without a corresponding student violation
  - storage growth > 100 MB / hour
  - log growth > 50 MB / hour

What to watch IN THE COMMAND CENTER while the load runs:
  active_sessions          should equal #VUs minus ramp-up window.
  failed_requests_per_hour stays at 0.
  events_per_minute        ~250 / minute at 50 VUs (roughly 5/student/min).
  auto_submit_breakdown    must be empty.
  Any non-empty alerts panel = test FAIL.
