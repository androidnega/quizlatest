 PHASE 1 - STUDENT CHEATING SIMULATION
================================================================================

ATTEMPTED ATTACKS AND OUTCOMES

A.1  Disable JavaScript on the take page
     Bypass: NO
     Why   : Server timer is computed from exam_sessions.start_time and
             exam.duration_minutes. Without JS the student cannot save
             answers, fire heartbeats, or run proctoring; the cron
             "exam:pause-stale-sessions" pauses them after 120s of no
             last_seen_at; on resume the elapsed pause is added to
             accumulated_pause_seconds so the time budget is preserved.
             They cannot gain extra writing time, only delay submission.

A.2  Modify the timer in DevTools / localStorage
     Bypass: NO
     Why   : ExamSessionTimer is server-only. The client renders a
             countdown but the server independently calls
             autoExpireIfTimedOut() on every endpoint and submits when
             remaining<=0 regardless of what the client believes.

A.3  Block /save-answer to skip persistence
     Bypass: NO
     Why   : Unanswered questions still receive 0 points via
             ExamAnswerSynthesisService::ensureEveryQuestionHasAnswer(),
             which runs inside submit() before grading.

A.4  Block /heartbeat
     Bypass: NO
     Why   : Same as A.1 - leads to a paused session, not extra time.

A.5  Replay a saved /submit payload
     Bypass: NO
     Why   : submit() now does an atomic compare-and-swap: only the
             first request that flips status from active/paused to
             submitted wins; replays return immediately.

A.6  Multiple tabs open to game tab-switch counter
     Bypass: NO
     Why   : tab_switch_count is server-side; ingestTabSwitch()
             increments atomically; visibilitychange is per-tab so each
             tab generates its own strikes.

A.7  Multiple devices with same student account
     Bypass: NO
     Why   : ExamEntryPipelineService blocks a second start when an
             active/submitted session exists; ExamRedisService.
             acquireSessionStartLock() also rejects a parallel start
             via Cache::add() with NX TTL.

A.8  Spoof "phone NOT detected"
     Bypass: PARTIALLY (camera-based detection is one-sided)
     Why   : The system can only react to phones it sees. A student
             concealing a phone behind the camera cannot be caught by
             coco-ssd. This is a fundamental limitation of camera-based
             enforcement, not a bug. Mitigated by examiner review of
             flagged_for_review sessions.

A.9  Spoof a synthetic "tab_switch" event for someone else
     Bypass: NO
     Why   : authorizeStudentProctoringSession enforces
             student_id === auth user id; metadata session/student/exam
             ids are validated against the bound ExamSession.

A.10 ** HIGH ** Spam /proctoring-overlay/clear to dismiss the
     blur after a face-obstruction strike, then keep cheating
     Bypass before fix: YES
     Bypass after  fix: NO
     Why before    : ingestOverlayResolved() unconditionally cleared
                     proctoring_blur_active. A student in DevTools
                     could POST { event_type:'proctoring_overlay_resolved',
                     metadata:{resolved_reason:'student_cleared'} }
                     to bypass camera enforcement indefinitely.
     Fix applied   : The orchestrator now counts every overlay
                     dismissal in violation_events. After
                     MAX_STUDENT_OVERLAY_DISMISSALS (=3) attempts the
                     session escalates to autosubmit with reason code
                     "overlay_dismiss_abuse" and exam_status is set to
                     flagged_for_review. The student cannot continue
                     answering. clearProctoringOverlay() now mirrors
                     this submit path.

A.11 Hidden iframe to load the exam in a sneaky way
     Bypass: NO
     Why   : Routes are auth-gated; CSRF middleware rejects cross-origin
             POSTs without a fresh token; the take page is rendered
             fresh on each load.

A.12 Modify violation_count in browser
     Bypass: NO
     Why   : Counts are mutated server-side via
             $session->increment(...); the client value is read-only.

 PHASE 2 - CONCURRENT REQUEST ATTACK
================================================================================

  100 parallel POST /save-answer for one (session, question)
    Outcome    : ONE row in exam_session_answers (UNIQUE constraint on
                 (exam_session_id, question_id) plus the upsert).
    Race risk  : MEDIUM (M1) - the SELECT-then-upsert is non-atomic. A
                 stale request that arrives AFTER a fresher one can
                 overwrite the fresher answer because the stale-revision
                 check runs in PHP, not in SQL. Practical impact is
                 narrow because (a) same student = same session file =
                 PHP-FPM serialises them, and (b) the next save fixes
                 the value. No data loss in normal operation.
    Suggested  : append "AND client_revision <= ?" to the upsert WHERE
                 clause for full atomicity.

  50 parallel POST /submit for the same session
    Outcome    : ONE submission, ONE Result row, ONE evaluation pass.
    Reason     : ExamSessionSubmissionService::submit() now uses an
                 atomic UPDATE WHERE status != 'submitted'; only the
                 first caller proceeds, others see affected_rows=0
                 and return immediately.

  Heartbeat flood (1000 req/min from one student)
    Outcome    : Just refreshes last_seen_at. No DoS amplification (1
                 SELECT + 1 UPDATE per call). However there is no
                 route-level throttle on /heartbeat (M4); a malicious
                 client could keep an FPM worker busy.
    Suggested  : add throttle:60,1 to /heartbeat,/state.

  /state polling flood
    Same observation as heartbeat. Server cost is 3 queries per call.

 PHASE 3 - NETWORK FAILURE TEST
================================================================================

  Disconnect for 30 seconds
    Frontend  : axios catches network errors; banner shows "saving"
                state until reconnect. ProctoringEventBatcher caches
                events in memory and flushes on reconnect.
    Backend   : last_seen_at not refreshed; cron does NOT pause yet
                (threshold 120s). Timer keeps ticking server-side.
    Verdict   : student loses 30s of effective writing time but no
                data loss. Acceptable.

  Disconnect for 5 minutes
    Frontend  : same; on reconnect the next /state poll returns the
                authoritative timer.
    Backend   : after 120s the cron paused the session. On reconnect
                the student calls /resume which adds the elapsed pause
                to accumulated_pause_seconds, preserving their original
                budget. Timer keeps the right amount of time.
    Verdict   : works as designed.

  Disconnect for 15 minutes
    Frontend  : same.
    Backend   : session paused for 15 minutes. Resume adds 15 min to
                accumulated_pause_seconds.
    Verdict   : MEDIUM (M2) - there is NO upper bound on pause duration.
                A student could pause indefinitely and resume hours
                later. The time BUDGET is correctly preserved (no extra
                writing time), but they get unbounded wall-clock time
                AFTER pausing - long enough to research answers offline
                before answering. This is an exam-integrity gap.
    Suggested : auto-submit any session that has been paused longer
                than max_pause_minutes (suggest 10 min, configurable).

 PHASE 4 - DATABASE FAILURE TEST
================================================================================

  Temporary outage during /save-answer
    Behaviour : QueryException bubbles up; Laravel returns 500. Client
                retries on next save. The unsubmitted answer is lost
                until the next save. No corruption.

  Temporary outage during /submit (after status flip but before
  ResultFinalization)
    Behaviour : MEDIUM (M3) - submit() is not wrapped in DB::transaction.
                The atomic CAS already flipped exam_sessions.status to
                'submitted'. If the connection then drops, ensureEvery
                QuestionHasAnswer / evaluate / syncAfterSubmission
                aborts. The session is "submitted" with no Result row
                and partially evaluated answers. A retry returns
                immediately because of the atomic guard - the Result
                will never be written.
    Suggested : wrap the post-CAS pipeline in DB::transaction(), and on
                failure catch QueryException and re-attempt result
                finalisation. ResultFinalizationService is already
                idempotent (updateOrCreate on user+quiz), so a retry
                hook is safe.

  Slow queries (e.g. proctoring_events insert taking >1s)
    Behaviour : Request blocks the FPM worker. With 12 saveAnswer per
                minute per student and one slow query per request,
                250 students = potential FPM exhaustion.
    Mitigation: every hot path now hits an index (Phase 1 of the
                Production Readiness report). No slow query identified.

  Deadlock during concurrent submit
    Behaviour : Atomic UPDATE on a single row cannot deadlock with
                itself. updateOrCreate inside ResultFinalization could
                in theory deadlock with another writer on (user_id,
                quiz_id). MariaDB will pick a victim and return
                Deadlock; Laravel re-runs the closure once via the
                default transaction retry. Acceptable.

  Lost connection between heartbeats
    Behaviour : QueryException -> 500. Client treats as transient and
                retries on the next interval. No corruption.

 PHASE 6 - DISASTER RECOVERY
================================================================================

  Apache / PHP-FPM restart during exam
    Submit in flight  : if killed mid-pipeline AFTER the atomic CAS,
                        same gap as DB outage (M3 above) - session is
                        submitted but has no Result. Examiner manual
                        re-finalisation needed.
    Save in flight    : answer not persisted; client retries on
                        next save. No data loss.
    Heartbeat lost    : not material; next heartbeat refreshes
                        last_seen_at. If gap >120s the cron pauses
                        the session and the student resumes seamlessly.

  Cache clear during exam (`php artisan cache:clear`)
    Impact : every cached lookup (academic year, term, classroom_year,
             practice streak, exam config) recomputes on next access.
             First request after the clear is slower by ~50-100ms.
             No data loss. ExamRedis cache fallbacks are recomputed
             from DB.

  Session cleanup (`storage/framework/sessions/* removed`)
    Impact : every authenticated user logs out. Students must re-login
             via OTP. Their exam_sessions row is untouched - they can
             continue where they left off (timer paused at most for
             120s). RECOMMEND NEVER doing this during an exam window.

  Deployment during exam
    Risks  : (1) if you restart FPM, in-flight requests die (mostly
             benign; see above). (2) if you change Quiz attributes
             mid-exam, students may see new copy on next state poll.
             (3) if you run migrate while a session is active,
             schema changes can break a request.
    Recommended: deploy outside exam windows. If unavoidable, use
             cPanel "graceful" reload (kill old workers only after
             they finish). Schedule migrations after sessions close.

3. REMAINING VULNERABILITIES (none CRITICAL or HIGH)

   MEDIUM
     M1  saveAnswer SELECT-then-upsert race could let a stale request
         overwrite a fresher answer in narrow concurrency windows.
         Practical impact bounded by FPM serialisation per session
         file; recoverable on next save.
     M2  No max-pause SLA. A paused session can sit for hours; the
         time budget is preserved, but the wall-clock window allows
         offline research.
     M3  ExamSessionSubmissionService::submit() lacks transactional
         wrapping. A connection drop between status flip and result
         finalisation leaves a session "submitted" with no result row.
     M4  No route-level throttle on /heartbeat, /state, /save-answer,
         /proctoring-events, /proctoring-events/batch, /submit.

   LOW
     L1  Tab-switch false-positive risk from OS notifications and
         browser permission dialogs (rare in practice).
     L2  Wiping storage/framework/sessions during an active exam
         logs every student out mid-attempt. Operational discipline
         required, not a code bug.

   - All five active rules (tab switch, phone, multiple faces, face
     missing, face obstruction) are calibrated conservatively (see
     Phase 5). Every auto-submit lands as exam_status=submitted_held
     (NOT submitted) so an examiner reviews before releasing the
     score - a student wrongly auto-submitted can have the result
     released or the session invalidated for retake.

5. RECOVERY RISKS
   - M3 above: rare but real. Mitigation: examiner "force re-finalise"
