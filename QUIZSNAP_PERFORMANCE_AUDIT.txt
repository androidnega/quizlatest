================================================================================
SECTION 0 — EXECUTIVE SUMMARY
================================================================================

QuizSnap is functionally complete (LMS + assessment engine + browser proctoring
+ assignment workflow + analytics) but its current configuration is the single
biggest reason it crashes under load. The application code is mostly sound; the
problem is that EVERY hot path (heartbeats, state polling, answer saves,
proctoring events, sessions cookies) is funneled through MySQL because the
.env disables caching/sessions/queues to anything except the database.

On a 500-student exam this means a single MariaDB instance on shared cPanel
must absorb roughly:

    - 3,000 session writes per minute      (Laravel session middleware)
    - 6,250 SELECT/INSERT per minute       (state polling every 8s/12s)
    - 1,200 heartbeat writes per minute    (every 25s)
    - 2,000-6,000 proctoring inserts/min   (face/phone/tab events)
    - 4,000 answer save UPSERTs per minute (every ~25s of typing)

Total: 16,000-22,000 queries/minute on the same MySQL user that cPanel limits
to ~75 concurrent connections. This is why your server returns "Resource
Exhausted" and 503/500 errors during exams.

The good news: the architecture has all the right primitives. Almost every
issue is fixable WITHOUT a VPS, WITHOUT Redis, and WITHOUT rewriting features.

The fastest wins (and exactly what to change first) are:

    1. Move SESSION_DRIVER from "database" to "file"        (eliminates ~50% of DB load)
    2. Move CACHE_STORE from "database" to "file"           (eliminates per-request cache contention)
    3. Fix .env: APP_DEBUG=fasle (typo) -> APP_DEBUG=false  (DEBUG mode is currently ON)
    4. LOG_LEVEL=debug -> LOG_LEVEL=warning                 (stops growing storage/logs/laravel.log every request)
    5. LOG_STACK=single -> LOG_STACK=daily + 7 day retention (single file currently grows unbounded)
    6. Add 6 missing indexes (Section 2)                    (cuts query time 5-30x for examiner views)
    7. Eliminate the 9-15 ->fresh() round-trips per request in
       ExamSessionController and ProctoringOrchestratorService (Section 3)
    8. Reduce client polling cadence (8s/12s) and rely on the existing
       proctoring batch endpoint (4.5s) for state hints (Section 7)
    9. Disable broadcast() calls when BROADCAST_CONNECTION=log (currently
       Reverb events are dispatched through the log driver every event - a
       hidden cost) (Section 6)
   10. Adjust scheduler ("schedule:run every minute" must run via cPanel
       cron; "queue:work" must NOT be relied upon - move all queueable work
       to "sync") (Section 9)

If priorities 1-3 alone are applied, your server will stop crashing for
50-150 concurrent students with no other changes. Priorities 1-10 raise the
realistic ceiling to 400-600 concurrent students on a typical cPanel plan
(2 vCPU / 4 GB RAM / shared MySQL).

1.3 CRITICAL CODE-LEVEL ISSUES (file-by-file, line-referenced)
--------------------------------------------------------------------------------

[ISSUE 1.3.1]  Excessive ->fresh() round trips in ExamSessionController
File: app/Http/Controllers/ExamSessionController.php
Lines: 100-114, 116-145, 174-258, 259-274, 290-368, 370-464

Every state, heartbeat, answer save, proctoring event, and resume call hits
the database 3-6 EXTRA times because the controller calls ->fresh() to reload
the row after every update. Example saveAnswer at line 174:
  - Question::query()->where->first()                                (1)
  - ExamSessionQuestion::query()->where->first()                     (2)
  - ExamSessionQuestion::query()->where->exists()                    (3)
  - ExamSessionAnswer::query()->where->first()                       (4)
  - ExamSessionAnswer::updateOrCreate                                (5)
  - $examSession->forceFill->save()                                  (6)
  = 6 DB round trips per typed character (debounced to ~1.6s)

state() at line 100 already does 4 round trips on a single GET.
heartbeat() at 259 does at minimum 3.

For 500 students saving an answer every ~25 seconds and polling state every
8 seconds, this multiplies into ~2,400 + 18,750 = 21,150 round-trips/minute
to MySQL per exam.

FIX:
  - Replace `->fresh()` after `->save()` with the in-memory model when the
    update payload is known (most cases).
  - Where you must re-read, read only the columns needed:
      $examSession->refresh(); // reuses the same object
  - Combine question lookup + session-question lookup into a single JOIN.
  - Remove the redundant `ExamSessionQuestion::exists()` call by relying on
    the result of the first lookup ($sessionQuestion === null && pool empty).
Estimated saving: 50-65% fewer queries on /answers and /state.


[ISSUE 1.3.2]  ProctoringOrchestratorService re-loads the session repeatedly
File: app/Services/ProctoringOrchestratorService.php
Lines: 277, 283, 326-430, 496-592, 601-683, 692-785, 792-862, 872-932, 939-1009

Every variant of ingest*() begins with:
    $session = $examSession->fresh() ?? $examSession;
then later:
    $session = $session->fresh() ?? $session;
and finally:
    $out = $session->fresh() ?? $session;

That is 3 re-reads of `exam_sessions` on every proctoring event. Combined
with eventBatcher (max 14 events per request) the controller method
logProctoringEventBatch can produce 14 * 3 = 42 SELECTs from exam_sessions
in a single HTTP request, plus 14 INSERTs into proctoring_events, plus
14 UPDATEs to exam_sessions.

FIX:
  - In each ingest method, mutate the in-memory $examSession copy then call
    a single ->save() at the end. Do NOT re-read after each update.
  - Move the update payload assembly to a single $session->update([...]) at
    the bottom of each method.
  - For batch: pass the row through; do not re-fetch between events.

Estimated saving: 70% fewer SELECTs against exam_sessions during exam
proctoring.


[ISSUE 1.3.3]  broadcast() always fires - even when BROADCAST_CONNECTION=log
File: app/Services/ProctoringOrchestratorService.php (lines 1190, 1202, 1222,

File: app/Services/ProctoringGlobalControlService.php (line 108)

The codebase calls broadcast(...) on every risk transition, every warning,

[ISSUE 1.3.4]  StudentDashboardDigestService::practiceStreakDays runs 60 queries
File: app/Services/StudentDashboardDigestService.php (lines 201-219)

The method literally loops 60 times and runs a SELECT EXISTS for each day:

        $has = PracticeAttempt::query()->...->whereBetween([...])->exists();
    }

If practice is enabled, every dashboard hit costs 60 DB round trips. With
500 students reloading the dashboard occasionally, this is up to 30,000
queries/minute that do not need to exist.

FIX (rewrite once, in PHP, from a single grouped query):

        ->where('student_id', $user->id)
        ->whereNotNull('submitted_at')
        ->where('submitted_at', '>=', now($tz)->subDays(60)->startOfDay())

        ->groupBy('d')
        ->pluck('d')
        ->all();
    // walk back day-by-day in PHP, count consecutive days.

Estimated saving: 60 queries -> 1 query per dashboard hit.

4.2 PROBLEM AT SCALE
--------------------------------------------------------------------------------
With 500 concurrent students and an exam page that issues:
   - 1 fetchState every 8s     = 7.5 req/min
   - 1 heartbeat every 25s     =  2.4 req/min
   - 1 batched proctoring/4.5s = 13   req/min
   - typing -> answer save     = 1-3  req/min
   = ~25 requests/min/student * 500 = 12,500 req/min
Each request opens the Laravel session middleware which:
   1. SELECT * FROM sessions WHERE id=?        (one row per HTTP req)
   2. Decrypt+unserialize payload (longtext)
   3. Call StartSession::handle()
   4. UPDATE sessions SET payload=?, last_activity=? WHERE id=?
That's 25,000 hot writes/min on ONE row per user. The lottery sweeper
also runs DELETE FROM sessions WHERE last_activity < ? on 2% of the same
hot path -> 250 sweeper deletes/min.

4.3 RECOMMENDATION: SESSION_DRIVER=file
--------------------------------------------------------------------------------
The "file" driver writes to storage/framework/sessions/<sha1-id>.
Filesystem writes on cPanel are local SSD with ext4/XFS - much faster
than MySQL UPDATE under contention.
Pros for cPanel:
  - No DB load for session at all
  - Per-request write is a tiny <8 KB file
  - cPanel "Disk Usage" rarely complains; storage/framework/sessions
    rotates via Laravel's lottery
Cons:
  - One server only (you have one - this is fine on shared hosting)
  - File handle pressure if you exceed ~10,000 active sessions (mitigate
    via SESSION_LIFETIME shortened)

Verdict: use file. Cookie driver is NOT acceptable - student answers
trigger session writes during exam, and cookie driver caps at 4 KB and
ships the entire payload on every request.

[HOST 9.1.1] Laravel Reverb / WebSockets
Reverb requires a long-running PHP/Swoole process. cPanel doesn't allow
this. Currently you have BROADCAST_CONNECTION=log in .env - good.
Action: Document explicitly that broadcast events are no-ops in production
and remove broadcast() calls from hot paths (heartbeat).

[HOST 9.1.2] Queue workers (database driver)
.env QUEUE_CONNECTION=database. Without `php artisan queue:work` running
under Supervisor, queued jobs accumulate in `jobs` table and never run.
Found jobs that go via dispatch():
  - app/Jobs/...   (search for class extending ShouldQueue)
Recommendation:
  a) For mail/notification: dispatch with QUEUE_CONNECTION=sync so they
     run inline (acceptable when send time < 200ms).
  b) For long jobs (PDF result generation): use cPanel cron with
     "php artisan queue:work --once --stop-when-empty --tries=3" every
     5 minutes. This polls + drains and exits.
  c) Remove dispatch() from request hot paths entirely.

[HOST 9.1.3] Scheduled tasks
You ARE using Laravel Scheduler (routes/console.php). On cPanel set:
  * * * * * cd /home/USER/quizsnap && php artisan schedule:run >> /dev/null 2>&1

[HOST 9.1.4] OPcache and JIT
Most cPanel hosts have OPcache available but with `opcache.validate_timestamps=1`
On deployment run `php artisan opcache:clear` (requires
`appstract/laravel-opcache` package or call `opcache_reset()` manually
via a deploy script).

[HOST 9.1.5] Symbolic links
public/storage -> storage/app/public is a symlink. Some cPanel servers
disallow symlinks via Apache. If after `php artisan storage:link` the
public files 404, use a stub htaccess redirect or copy files instead.

[HOST 9.1.6] putenv / proc_open
Some shared hosts disable proc_open (DeepSeekAiService uses Guzzle/HTTP -
no proc_open needed - good).

9.2 FILES THAT NEED TWEAKS FOR SHARED HOSTING
--------------------------------------------------------------------------------
- bootstrap/app.php - check that withMiddleware() doesn't register Telescope
  or Debugbar in production (line 30-50, verify).
- app/Providers/AppServiceProvider.php - ensure all *Service*::singleton()
  bindings are deferred where possible (DeferrableProvider).
- config/queue.php - default driver should be 'sync' for cPanel.
- public/.htaccess - add cache headers for static assets:
    <FilesMatch "\.(jpg|jpeg|png|webp|css|js|woff2|ico)$">
        Header set Cache-Control "public, max-age=31536000, immutable"
    </FilesMatch>

9.3 RAM PROFILE
--------------------------------------------------------------------------------
Each PHP-FPM worker on cPanel = 35-65 MB resident. With max_children=10
(typical shared) you get ~10 concurrent requests. With 500 students each
sending 1.5 req/sec you need ~750 RPS - far above the 10 concurrent limit.
This means requests queue. cPanel RAM is the hard ceiling.

WORKAROUND:
  - Reduce queries per request (Sections 1-3) so each request finishes
    in <80 ms instead of 250-500 ms.
  - At 80 ms/request, 10 workers = 125 RPS sustained - enough for 500
    students with 0.25 RPS each (which is what we get after the AJAX
    reductions in Section 7).
  - Beyond that: add a CDN in front of all static assets (Cloudflare
    free tier serves /build/*, /images/*, /favicon.ico) so PHP only
    handles dynamic /student/exam/* traffic.
