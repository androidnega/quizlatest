 PHASE 7 — TAB-SWITCH 3-STRIKE AUTO-SUBMIT
================================================================================

VERIFIED — already fully implemented.

Frontend (resources/js/studentExamRuntime.js):
  - Listens for `visibilitychange` AND `window.blur/focus` (defence in
    depth — some OSes only fire one).
  - State machine: `tabSwitchDocWasHidden` flag flipped on hidden, then
    on next visible/focus → `recordTabSwitchStrike()` posts a
    `tab_switch` proctoring event.
  - In-flight guard `tabSwitchRecordInFlight` prevents double-records.

Backend (ProctoringOrchestratorService::ingestTabSwitch):
  - Increments `tab_switch_count` atomically with `->increment()`.
  - Strike 1 → action='warn',     risk='warning'
  - Strike 2 → action='warn',     risk='suspicious', exam_status='flagged_for_review'
  - Strike 3 → action='autosubmit', risk='locked'

Persistence:
  - Per-event row in `proctoring_events` (with index proctoring_events_session_id_idx).
  - `tab_switch_count` and `risk_state` on `exam_sessions`.

Auto-submit dispatch (ExamSessionController::logProctoringEvent):
  - When decision['auto_submit'] === true →
    `sessionSubmission->submit($examSession, 'submitted_held',
                              'violation_threshold', 'tab_switch_locked')`.

 PHASE 9 — CAMERA MESH (FACE / FACEMESH) ENFORCEMENT
================================================================================

VERIFIED — face missing detection works; minor enhancement recommended.

Frontend (resources/js/proctoringRuntimeEngine.js):
  - MediaPipe FaceLandmarker + PoseLandmarker (dual mesh).
  - faceLandmarker.detectForVideo() runs every faceIntervalMs (12 s
    capable / 18 s low-end).
  - faceMissStreak counter increments on each empty result.
  - On streak ≥ 2 AND not already open → `emit face_missing(faces=0,
    source: interval_debounced)`.
  - Pose landmarks check wrist-near-face → emits face_obstructed when
    distance < 0.12 (normalized).

Backend (ProctoringOrchestratorService):
  - face_missing → routed to ingestLegacyWeightedEvent (uses
    violation_weights['face_missing'] = 10 toward the weighted
    risk-state ladder).
  - face_covered/face_obstructed/face_not_clear → ingestFaceObstruction
    with strike counter.

Behaviour today:
  - Repeated face_missing accumulates weighted score.
    score ≥ 30 → warning, ≥ 50 → suspicious, ≥ 70 → critical,
    ≥ 90 → locked + auto-submit.
  - With the default weight of 10, that is 9 missed-face events to
    auto-submit — roughly 9 × 12 s = 108 s of lost camera, minus the
    streak-of-2 debounce. Reasonable.

OPTIONAL ENHANCEMENT (NOT a blocker):
  Add an explicit face_missing 3-strike branch parallel to tab_switch:
    Strike 1 → warn
    Strike 2 → severe warn + flag_for_review
    Strike 3 → auto-submit
  This makes the rule symmetric and explainable to students. Track in
  follow-up issue; current weighted-score path already auto-submits
  egregious cases.

NO DIFF NEEDED for production safety. (Filed as OPTIONAL.)

 PHASE 10 — AUTO-SUBMISSION HARDENING (exactly-once)  ← FIX APPLIED
================================================================================

CRITICAL FIX APPLIED in app/Services/ExamSessionSubmissionService.php

Problem:
  Previous logic was:

      public function submit(...): void {
          if ($examSession->status === 'submitted') {
              return;                                  // ← in-PHP guard only
          }
          ...
          $examSession->update($update);               // ← non-atomic
          ... answer synthesis, evaluation, result finalisation ...
      }

  Two concurrent paths (e.g. timeout + 3rd-strike tab switch + manual
  POST /submit racing through different PHP-FPM workers) could both pass
  the in-PHP guard, both call ->update(), and both run the entire
  expensive finalisation pipeline — producing duplicated result rows in
  the worst case (mitigated only because Result::updateOrCreate happens
  to be idempotent on (user_id, quiz_id)).

Fix:
  Replaced with an atomic compare-and-swap update:

      $affected = DB::table('exam_sessions')
          ->where('id', $examSession->id)
          ->where('status', '!=', 'submitted')
          ->update($update);

      if ($affected === 0) {
          return;                                      // someone else won
      }

  Only the first request that flips status away from 'submitted' wins;
  every other concurrent caller sees affected_rows = 0 and returns
  immediately, eliminating duplicate answer-synthesis, duplicate
  evaluation, duplicate Result writes, and duplicate broadcast events.

Verification matrix — every path that calls submit():

  Path                                          | Caller                                       | Hardened?
  ----------------------------------------------+----------------------------------------------+-----------
  Manual POST /exam-sessions/{id}/submit        | ExamSessionController::submit (line 711)     | YES
  Tab switch ≥3 strikes                         | logProctoringEvent (line 379, 473)           | YES
  Phone detected                                | logProctoringEvent (line 379, 473)           | YES
  Multiple faces ≥30 s                          | logProctoringEvent (line 379, 473)           | YES
  Screenshot/screen-record auto-submit          | logProctoringEvent (line 379, 473)           | YES
  External-display violation                    | logProctoringEvent (line 379, 473)           | YES
  Weighted-score threshold (90)                 | logProctoringEvent (line 379, 473)           | YES
  Timeout                                       | autoExpireIfTimedOut → submit (line 36)      | YES (via submit)
  Examiner force-submit                         | ExamSessionController::forceSubmit (line 722)| YES (via submit)

Test results after fix:

  → 93 / 93 passing (409 assertions, 8.6 s)
