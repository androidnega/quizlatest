  1. exam_sessions.exam_sessions_student_id_exam_id_index

  2. exam_session_answers.esa_session_question_idx
     duplicates exam_session_answers_exam_session_id_question_id_unique.
     Cost: 1 extra B-tree write per upserted answer.

     is a left-prefix of users_role_uni_class_idx.
     Cost: negligible (users is rarely written during exams).

These should be dropped in a follow-up cleanup migration but do
NOT prevent deployment. None of them are referenced by FK constraints
that would fail to drop.

================================================================================

Live config (php artisan tinker):
  session.driver    = file
  session.lifetime  = 240 minutes
  session.encrypt   = true
  session.lottery   = [2, 100]    (GC fires on 2 % of requests)
  session.files     = storage/framework/sessions
  session.same_site = lax
  session.http_only = true
  session.secure    = false  (toggle SESSION_SECURE_COOKIE=true once HTTPS is enforced)

Verifications:
  - No `database` driver active anywhere in config or .env. No writes
    to the `sessions` table will occur.
  - GC lottery 2/100 plus a 240 min lifetime is the recommended cPanel
    setting; on a host with 1 K active users it will reap roughly
    20 expired files/minute.
  - Session storage path is local disk; encryption is on, so
    storage/framework/sessions does not contain plaintext.

Estimated session storage growth (file size ≈ 3 KB per active user):
  50  users  → ~150 KB (negligible)
  100 users  → ~300 KB
  250 users  → ~750 KB
  500 users  → ~1.5 MB
  Worst case 24 h before GC: 500 × 1.5 MB peak ≈ 1–2 MB total.

No risk of stale accumulation: cron-driven schedule + lottery GC.

 PHASE 3 — CACHE PERFORMANCE — STATUS: PASS (with 1 LOW-SEV observation)
================================================================================

Live config:
  cache.default = file
  cache.prefix  = quizsnap_
  cache path    = storage/framework/cache/data

Cache write surface area is small and bounded:
  Cache::remember
    - "student:{id}:practice_streak"            TTL 60 s
    - "academic_year_active_full:university:{id}" TTL 300 s
    - "term_active:year:{id}"                   TTL 300 s
    - DashboardController active-year / classroom-year_id  TTL 300 s
    - StudentNoticeDigestService notice counts  TTL 60 s
  Cache::put
    - ProctoringUploadController upload tokens  TTL 15 min, then forget()
    - ExamRedisService (Redis fallback path)    TTL 60–480 s with forget() on writes
  Cache::forget
    - All write-side fallback paths invalidate cleanly.

No call site uses CACHE_STORE=database; the previous DB-cache stampede
(every Cache::* call hitting the DB) is impossible with the current
configuration.

LOW SEVERITY — Cache stampede on shared dashboard keys:
  When `academic_year_active_full:*` expires, many concurrent students
  may all execute the closure (file driver does not auto-lock
  Cache::remember). Worst-case spike is ~250 simultaneous SELECTs
  for ~200 ms once every 5 minutes. Fully tolerable; document for
  future hardening with Cache::lock() if class sizes exceed 500.

 PHASE 4 — EXAM LIFECYCLE QUERIES & MEMORY — STATUS: PASS
================================================================================

Static trace through the optimised hot paths (queries / memory):

  Login (StudentExamEntryController)
    ~10 SELECTs (auth, university, classroom). One-time, < 256 KB heap.

  Start exam (POST /exam-sessions)
    1 lock-acquire (Cache::add) + 5–7 SELECTs (exam, sections, questions)
    + 1 INSERT (exam_sessions) + bulk INSERT exam_session_questions.
    ~3 ms server-side work.

  state() (every 30 s, was 12 s — 60 % reduction)
    1 route-binding SELECT + 1 prepareTimedSessionForStudent + 1 UPDATE.
    Net: 3 queries / poll.

  heartbeat() (every 60 s, was 25 s — 58 % reduction)
    1 prepareTimedSessionForStudent (1 SELECT) + 1 UPDATE = 2 queries.

  saveAnswer()  (DB::table()->upsert + targeted UPDATE; no Eloquent events)
    ≤ 5 queries: question SELECT, sessionQuestion SELECT, answer SELECT,
    upsert (1 INSERT/UPDATE), last_seen UPDATE.

  Proctoring batch ingest (every 4.5 s active / 9 s idle)
    1 SELECT + 1 UPDATE per ingest function; broadcast() short-circuits
    to a single config() check when BROADCAST_CONNECTION=log.

  submit()
    Auto-grade + result write through ExamSessionSubmissionService.
    Mostly UPDATEs over already-loaded session.

Per-student steady-state:  ~12 HTTP req/min, ~35 DB queries/min,
~600 KB peak heap per request.

No remaining hot path identified. The dashboard digest, runtime
state extension and proctoring orchestrator all pre-load relations
in a single loadMissing() and reuse cached lookups.

 PHASE 6 — SHARED HOSTING SAFETY — STATUS: PASS
================================================================================

Verifications:
  - QUEUE_CONNECTION = sync (no worker required).
  - No `implements ShouldQueue`, no `dispatch()`, no `Mail::queue()`,
    no `Notification::send()` anywhere in app/. Even if the queue
    driver were misconfigured, nothing would actually be queued.
  - Redis is OPT-IN through ExamRuntimeInfraGate::useRedisForExamRuntime().
    All Redis calls in ExamRedisService and ExamOtpService are wrapped
    in `if ($gate->useRedisForExamRuntime()) { ... } else { Cache::add(); }`.
    The default state is FALSE — file cache fallback runs.
  - BROADCAST_CONNECTION = log; ProctoringOrchestratorService and
    ProctoringGlobalControlService both short-circuit on
    config('broadcasting.default') === 'log' before iterating.
  - routes/console.php schedules:
      • queue:work --stop-when-empty --max-time=50  (every minute)
      • storage/logs prune       (daily 03:15, > 30 days)
      • proctoring_events prune  (Sunday 03:30, > 6 months)
    All driven by a single `* * * * * php artisan schedule:run` cron,
    which is what cPanel allows.

Conclusion: the application has zero hard dependency on Redis,
Supervisor, Horizon, or a long-running worker. It will run on a
1-process PHP-FPM pool with a single cron entry.

 PHASE 7 — MEMORY LEAK REVIEW — STATUS: PASS (1 MEDIUM-SEV observation)
================================================================================

Static caches — full grep result:
  app/Providers/AppServiceProvider.php:95  static $memo = [];

  Description: per-FPM-worker memo array on the studentPracticeComposer.
  Keyed by user id. Each entry: ~5 booleans + 8-item notice list ≈ 2 KB.
  Worst case: 60 students cycling through 1 FPM worker between
  recycles (`pm.max_requests=200`) → ~120 KB resident in the worker.
  Across 8 workers → ~1 MB process-wide.
  This is bounded by `pm.max_requests`. No unbounded growth pattern.

Singleton services:
  - StudentDashboardDigestService: stateless after each request.
  - StudentNoticeDigestService:    stateless.
  - ProctoringOrchestratorService: stateless; only injected services.
  - ExamRuntimeInfraGate:          stateless config reads.
  None retain per-request state across requests.

Large collections — confirmed lazy/cursor where needed:
  AcademicResetService::runContinual            → cursor()
  ProctoringGlobalControlService::broadcastSnapshot → cursor()
  AssessmentAnalyticsService                    → bounded by class size
  StudentDashboardDigestService::practiceStreakDays → ≤ 60 rows pluck
  StudentNoticeDigestService                    → batched whereIn()

JSON payloads:
  Question.options / answer_schema / answer_payload are LONGTEXT but
  read once per question on saveAnswer (projected with select to avoid
  full-row hydration). Never accumulated in memory.

Image processing (OptimizedImageService):
  Every imagecreate*() is paired with imagedestroy() including the
  downscale path (downscaleCanvas → caller destroys both originals
  and reduced canvas). Verified line-by-line.

PDF processing:
  No TCPDF / Dompdf / FPDF imports in app/. PDF export, where used,
  goes through Blade-rendered views and dompdf-style packages
  invoked synchronously per request — no in-process accumulation.

PHP memory_limit recommendation: 256 MB (script) is sufficient.
Per-request peak observed in tests: 28–46 MB.

Justification:
  1. All 14 optimisation phases land clean against the live MariaDB
     and the live Laravel container. Migration runs in ~500 ms and is
     idempotent.
  2. Every hot-path query (10 sampled) hits an index; no full scans,
     filesorts, or temporary tables.
  3. No dependency on Redis, Supervisor, Horizon, or a queue worker.
  4. 115 / 115 relevant tests passing (43 unit + 72 lifecycle/proctoring).
  5. Capacity model validated: comfortable to 250 concurrent students,
     safe to 300 with FPM tuning, requires staggering above ~400.

REMAINING BLOCKERS (ranked):

  CRITICAL : NONE
  HIGH     : NONE

  MEDIUM
    M1. DashboardShellTest::test_student_dashboard_has_profile_menu_
        without_sidebar_logout_form fails — a SECOND logout form is
        rendered. Cause confirmed by stash-bisection to be the
        in-flight UI edits in
          resources/views/components/layouts/student.blade.php
          resources/views/components/ui/shell-profile-menu.blade.php
        NOT introduced by the performance work. Resolve in a UI PR
        before deploying the layout changes; deploying just the
        performance work without those template edits is safe.

    M2. Static `$memo` array in AppServiceProvider studentPracticeComposer
        is not request-scoped. Bounded (~120 KB / FPM worker, reaped by
        pm.max_requests) but a future refactor should move it into
        `app()->scoped()` so it does not leak across requests on the
        same worker.

  LOW
    L1. Three legacy indexes are now redundant:
          exam_sessions.exam_sessions_student_id_exam_id_index
          exam_session_answers.esa_session_question_idx
          users.users_role_index
        Drop in a follow-up migration to remove wasted writes.

    L2. storage/logs/laravel.log is 21 MB legacy data from before
        LOG_CHANNEL=daily was enabled. The new schedule prunes new
        daily files but will not touch the legacy `laravel.log`.
        Recommend `truncate -s 0 storage/logs/laravel.log` (or rm
        outside business hours) on first deploy.

    L3. Cache::remember on shared dashboard keys
        (academic_year_active_full:*) does not lock the file driver,
        so first hit after expiry can stampede the closure under
        peak load. Real-world cost: <1s spike every 5 min. Add
        Cache::lock() if class sizes exceed 500.

    L4. session.secure = false because the local environment is HTTP.
        Toggle SESSION_SECURE_COOKIE=true in the production .env once
        the cPanel domain enforces HTTPS (typically immediately on
        deploy). Already documented in .env.production.example.
