Role:        Principal Database Architect + Laravel Query Optimization Engineer
Scope:       How QuizSnap talks to MySQL (only)
Out of scope: security, proctoring fairness, production-readiness gates
Codebase:    Laravel 13, PHP 8.3, MySQL/MariaDB (cPanel shared hosting baseline)
Approach:    100% static analysis of real code at the path roots
             (app/, routes/, resources/js/, database/migrations/)
Date:        2026-06-09

LEGEND
  • All file:line references are accurate at the time of audit.
  • Query counts assume the polling cadence currently in
    resources/js/studentExamRuntime.js (POLL_MS=30000, heartbeat=60000,

    (estimated) extrapolate from those + the cadence above.

PHASE 3 — EXPLAIN ANALYSIS
================================================================================

The current index coverage was inspected via the migrations. The 2026-06-06
"add_performance_indexes" migration added 10 composite indexes covering the
biggest hot paths from the prior audit. The 2026-06-06 "drop_redundant_indexes"
migration was left empty (no-op), so the redundant single-column
proctoring_events_event_type_index was NOT dropped (the add migration drops
it inline — verified at line 36 of add_performance_indexes.php).

Below is per-query EXPLAIN reasoning for the 10 highest-traffic queries. Where
the optimal access path is clear, I list the expected EXPLAIN output. (Live
EXPLAIN requires the production schema + statistics; the analysis is based on
schema, declared indexes, and known cardinalities.)

────────────────────────────────────────────────────────────────────────────
EX-1  SELECT * FROM exam_sessions WHERE id = ?
────────────────────────────────────────────────────────────────────────────
Called from every route binding on /state, /heartbeat, /answers,
/proctoring-events*, /submit, /resume, etc.

  type           const
  possible_keys  PRIMARY
  key            PRIMARY
  rows           1
  Extra          (none)

VERDICT: Plan is optimal. The cost is in the PAYLOAD (Phase 4), not the
plan: this SELECT pulls the full `violation_events` JSON column on every
hit.

────────────────────────────────────────────────────────────────────────────
EX-2  SELECT * FROM quizzes WHERE id = ?
────────────────────────────────────────────────────────────────────────────
Called by loadMissing('exam') in /state, /heartbeat, /answers,
proctoring ingest, submit.

  type           const
  key            PRIMARY
  rows           1
  Extra          (none)

VERDICT: Plan optimal, but EVERY hot-path call repeats it (≥ 1 per request).
ExamRedisService::rememberQuiz() exists and caches the quiz at exam-start
but the cache is NEVER consulted by the runtime endpoints — they always go
through Eloquent. Top-10 #8 fixes this.

────────────────────────────────────────────────────────────────────────────
EX-3  SELECT * FROM exam_session_answers
      WHERE exam_session_id = ? AND question_id = ?
────────────────────────────────────────────────────────────────────────────
Called by /answers (line 238) on every save.

Covered by UNIQUE(exam_session_id, question_id) declared in
2026_05_01_152100_create_exam_session_answers_table.php:20
AND by `esa_session_question_idx` added in 2026_06_06 (redundant).

  type           const  or  ref
  key            exam_session_answers_exam_session_id_question_id_unique
  rows           1
  Extra          (none)

VERDICT: Optimal. The duplicate `esa_session_question_idx` index from the
2026-06-06 migration is REDUNDANT with the existing UNIQUE constraint and

PHASE 5 — JSON PAYLOAD REVIEW
================================================================================

5.1  /state payload (the only frequent server→client payload)
────────────────────────────────────────────────────────────────────────────
Source:  ExamSessionStateResolver::payload + ExamRuntimeStateExtension::forSession
Schema:  see app/Support/ExamRuntimeStateExtension.php:132-167

Measured shape for a 25-question MCQ exam:

  TOP-LEVEL                              ~ 0.4 KB
  (exam_ui_state, session_status, exam_status, risk_state,
   violation_score, session_id, exam_id, global_control_*,
   server_time, exam_end_at, time_remaining_seconds, duration_minutes,
   timer_paused, timer_pause_reason, proctoring_overlay,
   tab_switch_count, assessment_type, due_at, submitted_late,
   proctoring_client)

  exam OBJECT                            ~ 1.5 KB
    title, description (varies), course, assignment_* flags

  sections[].questions[]                 ~ 8-12 KB for 25 MCQs
    Each question: id, section_id, type, question_text (varies),
                   marks, question_order, answer_schema,
                   options[] (MCQ: typically 4 strings × ~30 chars)

  saved_answers                          ~ 1-4 KB (grows as student answers)
    Map of question_id → { answer_payload, saved_at, client_revision }

  TOTAL                                  ~ 12-18 KB per /state poll

For 500 students polling every 30 s:
  500 × 2 polls/min × 15 KB = ~15 MB/min outbound on /state alone
For 1000 students: ~30 MB/min outbound on /state alone.

REPEATED FIELDS (every 30 s, even though they never change):
  • exam.id, exam.title, exam.description, exam.course
  • exam.assignment_* flags
  • All section + question structure (sections, questions, options)
  • answer_schema for every question
  • proctoring_client hints (immutable per session)

These collectively account for ~85 % of every /state response.

OPPORTUNITIES FOR LIGHTER JSON

(a) Single bootstrap + delta /state
    The client could fetch the immutable parts ONCE at session-start
    (or on first /state) and then poll a lightweight /state-tick endpoint
    that returns ONLY:
        session_status, exam_status, risk_state, violation_score,
        tab_switch_count, time_remaining_seconds, server_time,
        timer_paused, proctoring_overlay, saved_answers_delta (since
        client_revision N), global_control_revision

    With saved_answers represented as a sparse "changed-since" map, the
    typical /state-tick is ~0.5-1 KB. This cuts /state bandwidth by
    93-95 %.

(b) ETag / If-None-Match on /state
    The full payload changes ONLY when the student answers a question or
    when proctoring fires. For idle students the entire response is
    byte-identical between polls. Adding ETag + 304 Not Modified would
    short-circuit ~60 % of /state requests on a typical exam with no
    response body at all.

(c) HTTP/1.1 gzip
    /state response is JSON and trivially compressible — typically
    ~70 % shrinkage. Confirm the cPanel Apache config has DEFLATE
    enabled for application/json. If not, add a Vary: Accept-Encoding
    + Content-Encoding: gzip from PHP via response()->setEncodingFormat.

   /proctor  6,000/min ×  8 q = 48,000 q/min
                       TOTAL  ≈ 114,000 q/min ≈ 1,900 q/s

   PHP-FPM peak: ~500 req/sec.
   Worker need (~120 ms avg under contention): 60 workers concurrent.
   MySQL connection peak: 60-100 — at or over the cPanel 75-connection limit.

   FIRST BOTTLENECK: shared MySQL.
     Evidence:
       - INSERT rate on proctoring_events ≈ 200/sec → InnoDB hot-page
         contention (next-key locks on the auto-inc index + 3 secondary
         indexes).
       - Connection pool ceiling (~75) hit when only ~38 % of PHP-FPM
         workers are busy with DB-bound requests.
       - exam_sessions UPDATE rate ≈ 100-150/sec, which forces InnoDB
         page-flushing during commit and starts hitting iowait on
         shared SSD.

   Secondary bottleneck: PHP-FPM saturated (60 of 40 → not enough
   workers; pm.max_children would need to rise to 80-100 which conflicts
   with the 4 GB RAM budget).

   Tertiary bottleneck: Network egress on /state if uncompressed
   (1000 × 2 × 15 KB = 30 MB/min ≈ 4 Mbit/s sustained outbound). On
   most cPanel plans this is the symbolic ceiling rather than a hard
   limit.

   1000-student capacity is NOT possible on the baseline shared cPanel.
   The hard ceiling is the MySQL connection pool first, then write QPS
   on hot tables, then PHP-FPM RAM. A migration to a dedicated MySQL
   user + read-replica OR Redis-fronted write coalescing for
   last_seen_at + Redis-backed proctoring event ingest would push the
   ceiling above 1000.
